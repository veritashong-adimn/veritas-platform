지시문

통역 상품의 언어 구조를 번역과 분리하여 “양방향 통역”을 지원하도록 개편한다.

현재 문제:

* 통역도 번역과 동일하게:
  출발언어 → 도착언어
  구조를 사용 중
* 실제 통역은 양방향 진행이 매우 많음
* 현재 구조로는:
  한국어 ↔ 영어
  형태의 실무 통역 표현이 어려움
* 통역 견적/상품 표현이 실제 운영 방식과 맞지 않음

목표:
번역과 통역의 언어 구조를 분리하고,
통역에서는 양방향 통역을 자연스럽게 처리 가능하도록 수정.

━━━━━━━━━━━━━━━━━━

1. 번역과 통역 언어 구조 분리
   ━━━━━━━━━━━━━━━━━━

[번역]
기존 구조 유지:

* 출발언어
* 도착언어

표시:
한국어 → 영어

유지.

━━━━━━━━━━━━━━━━━━
2. 통역 언어 구조 변경
━━━━━━━━━━━━━━━━━━

통역 유형에서는:
“출발언어 / 도착언어” 대신 아래 구조 사용.

필드:

* 언어 A
* 언어 B
* 통역방향

━━━━━━━━━━━━━━━━━━
3. 통역방향 필드 추가
━━━━━━━━━━━━━━━━━━

통역방향 옵션:

* 양방향 (기본값)
* A → B
* B → A

기본:
양방향 선택.

━━━━━━━━━━━━━━━━━━
4. 상품명 자동 생성 수정
━━━━━━━━━━━━━━━━━━

[양방향]
한국어 ↔ 영어 동시통역

[A → B]
한국어 → 영어 순차통역

[B → A]
영어 → 한국어 수행통역

형태로 자동 생성.

━━━━━━━━━━━━━━━━━━
5. 코드 생성 구조 수정
━━━━━━━━━━━━━━━━━━

양방향 통역 코드 예시:

IN-KO-EN-SIM-###
또는
IN-KOEN-SIM-###

중 하나로 정리.

표시에서는:
↔ 기호 사용.

━━━━━━━━━━━━━━━━━━
6. 통역 상품 리스트 표시 수정
━━━━━━━━━━━━━━━━━━

상품 카드에서:

기존:
한국어 → 영어

변경:
한국어 ↔ 영어

표시 가능하도록 수정.

━━━━━━━━━━━━━━━━━━
7. 통번역 유형 처리
━━━━━━━━━━━━━━━━━━

통번역 유형에서는:

* 번역 작업 시:
  출발언어 → 도착언어

* 통역 작업 시:
  양방향 구조 사용 가능

구조 유지.

━━━━━━━━━━━━━━━━━━
8. 기존 데이터 migration
━━━━━━━━━━━━━━━━━━

기존 통역 상품:

출발언어/도착언어 기반 데이터를:

* 언어 A
* 언어 B

로 migration.

기본 통역방향:
양방향 지정.

기존 데이터 깨지지 않게 처리.

━━━━━━━━━━━━━━━━━━

→ 사용

통역:
↔ 기본 사용

UI 혼동 없게 분리.

10. 통역 실무 표현 기준 반영
━━━━━━━━━━━━━━━━━━

실제 견적 표현 기준 적용:

예:

* 한↔영 동시통역
* 영↔일 순차통역
* 한→중 수행통역

등 자연스럽게 표시 가능해야 함.

11. E2E 최소 테스트 원칙 적용
━━━━━━━━━━━━━━━━━━

이번 작업은 중규모 구조 변경이므로:
과도한 전체 E2E 반복 금지.

원칙:

* 관련 파일 최소 탐색
* 핵심 시나리오만 테스트
* data-testid / aria-label 우선 적용
* 동일 원인 2회 실패 시 즉시 중단 후 보고
* 전체 로그인 플로우 반복 금지

━━━━━━━━━━━━━━━━━━
12. E2E 확인
━━━━━━━━━━━━━━━━━━

반드시 확인:

* 번역 구조 정상 유지
* 통역 양방향 정상 표시
* 통역방향 선택 정상 동작
* 상품명 자동 생성 정상
* 상품 코드 생성 정상
* 상품 리스트 표시 정상
* 기존 통역 상품 migration 정상
* 빌드 오류 없는지 확인

E2E 결과까지 보고.
