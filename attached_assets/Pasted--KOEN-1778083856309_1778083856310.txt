지시문

상품/단가 관리 구조를 장기 확장 가능한 구조로 전면 수정한다.

현재 문제:

* 서비스유형(번역/통역)과 상품유형(번역/통역)이 중복됨
* 언어쌍(KOEN 방식) 사전 생성 구조가 언어 확장 시 비효율적임
* 상품/통번역사/견적 구조가 서로 다르게 되어 있음

목표:

* 상품 구조 / 통번역사 구조 / AI 자동매칭 구조를 통일
* 언어쌍 사전 생성 방식 제거
* 상품유형 중심 구조로 재설계

━━━━━━━━━━━━━━━━━━

1. 서비스유형 제거
   ━━━━━━━━━━━━━━━━━━

상품 등록 상단의:

* 서비스유형
* 언어쌍
* 카테고리

영역 제거.

기존 서비스유형 기반 자동결정 로직 제거.

━━━━━━━━━━━━━━━━━━
2. 언어쌍 구조 제거
━━━━━━━━━━━━━━━━━━

현재:

* 한→영(KOEN)
* 영→한(ENKO)

사전 생성 방식 제거.

대신 아래 구조로 변경:

* 출발언어(sourceLanguage)
* 도착언어(targetLanguage)

드롭다운 방식 적용.

통번역사 상세 페이지와 동일 구조 사용.

━━━━━━━━━━━━━━━━━━
3. 상품유형 중심 구조로 변경
━━━━━━━━━━━━━━━━━━

상품유형을 최상위 기준으로 사용:

* 번역
* 통역
* 통번역
* 통역장비
* 프로젝트
* 교통비
* 식대
* 숙박
* 기타비용

━━━━━━━━━━━━━━━━━━
4. 대분류 자동 변경
━━━━━━━━━━━━━━━━━━

상품유형 선택 시
대분류 옵션 자동 변경.

예시:

[번역]

* 일반번역
* 전문번역
* 출판번역
* 영상번역
* 자막번역
* SW번역
* 기타번역

[통역]

* 동시통역
* 위스퍼링통역
* 순차통역
* 수행통역
* 미팅통역
* 전시회통역
* 화상통역
* 전화통역
* 기타통역

━━━━━━━━━━━━━━━━━━
5. 중분류 구조 추가
━━━━━━━━━━━━━━━━━━

대분류 선택 시
중분류 자동 변경.

예:

전문번역:

* 법률
* 의료
* 기술
* 금융
* 계약서
* 논문

━━━━━━━━━━━━━━━━━━
6. 코드 생성 방식 변경
━━━━━━━━━━━━━━━━━━

기존:
TR-KOEN-GEN-001

변경:
TR-KO-EN-GEN-001

또는:
TR-KO-EN-LAW-001

자동 생성 규칙 수정.

규칙:
[상품유형코드]-[출발언어]-[도착언어]-[대분류/중분류]-[번호]

예:
TR-KO-EN-LAW-001
IN-KO-EN-SIM-001

━━━━━━━━━━━━━━━━━━
7. languages 테이블 기준 통일
━━━━━━━━━━━━━━━━━━

언어는 독립 관리:

예:
ko
en
ja
zh

언어쌍 테이블(language_pairs)은 deprecated 처리하거나 제거 준비.

10. 기존 데이터 마이그레이션
━━━━━━━━━━━━━━━━━━

기존 KOEN 방식 데이터는:

* sourceLanguage
* targetLanguage

로 분리 저장하는 migration 작성.

예:
KOEN → ko + en

기존 상품 데이터 손실 없이 migration 수행.
