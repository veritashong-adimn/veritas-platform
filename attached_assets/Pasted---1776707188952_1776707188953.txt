지시문

목표:
정산 기능을 “수동 입력/수동 처리 화면”에서
“작업 완료 후 정산 후보가 자동 생성되고, 지급 가능 상태까지 자동 분류되는 구조”로 고도화한다.

이번 작업은
👉 재무·정산 영역의 고도화 1차이며,
“자동 송금”이 아니라 “자동 정산 준비”를 목표로 한다.

---

1. 범위 정의

이번 1차 범위:

* 작업 완료 시 정산 후보 자동 생성
* 지급 예정일 자동 계산
* 세금/지급 유형 자동 계산
* 정산 상태 자동 분류
* 예외 건은 검토 필요로 자동 분기
* 정산 대시보드/리스트에 자동 생성 결과 반영

이번 단계에서 제외:

* 실제 은행 자동송금
* 홈택스/세금계산서 자동 발행
* PayPal/API 직접 송금
* 회계 프로그램 외부 연동

---

2. 핵심 개념

정산 자동화는 아래 두 단계를 분리한다.

1. 정산 생성

* 시스템이 자동 수행

2. 정산 지급

* 관리자가 최종 확인 후 처리

즉,
시스템은 “보낼 돈을 자동 계산하고 준비”하고,
사람은 “최종 지급 승인”을 한다.

---

3. 자동 생성 트리거

아래 조건 중 하나를 기준으로 정산 후보를 자동 생성한다.

기본 트리거:

* task 상태가 completed 로 변경될 때

추가 조건:

* 해당 task에 배정된 통번역사가 존재
* 이미 settlement가 생성되어 있지 않을 것

중복 방지:

* taskId 기준 unique 체크
* 이미 생성된 정산이 있으면 재생성 금지

---

4. 자동 생성 대상 데이터

정산 생성 시 아래 정보를 채운다.

기본 필드:

* taskId
* projectId
* translatorId
* translatorName
* grossAmount (원금액)
* settlementType
* taxType
* withholdingRate
* withholdingAmount
* vatAmount
* netAmount (실지급액)
* payoutDueDate
* status
* reviewReason (nullable)
* paymentMethod
* bankInfoSnapshot 또는 payoutAccountSnapshot
* createdAt

주의:
정산 생성 시점의 계좌/지급정보는 snapshot으로 저장한다.
나중에 통번역사 정보가 바뀌어도 기존 정산 내역은 유지되어야 한다.

---

5. 정산 유형 자동 분류

통번역사/작업/계약 정보 기준으로 정산 유형을 자동 분류한다.

최소 분류:

1. WITHHOLDING_3_3

* 일반 프리랜서
* 원천세 3.3%

2. VAT_INVOICE

* 세금계산서 발행 대상
* 부가세 별도 처리 가능

3. OVERSEAS_REMITTANCE

* 해외 송금 대상
* 수동 검토 필요

4. OTHER_REVIEW

* 자동 분류 불가
* 검토 필요

자동 판정 기준은 현재 DB에 있는 필드 기준으로 우선 구현하고,
정보가 부족하면 pending_review 로 보낸다.

---

6. 지급 예정일 자동 계산 규칙

사용자 회사의 현재 규칙을 반영한다.

기본 규칙:

* 작업 완료일이 1일~15일 → 해당 월 말일 지급
* 작업 완료일이 16일~말일 → 다음 달 15일 지급

예:

* 4월 10일 완료 → 4월 30일 지급 예정
* 4월 21일 완료 → 5월 15일 지급 예정

주의:

* 말일 계산 정확히 처리
* 월말/윤년 등 date 계산 오류 없게 구현
* timezone은 한국 기준으로 처리

---

7. 세금 계산 규칙

1) WITHHOLDING_3_3

* withholdingRate = 3.3%
* withholdingAmount = grossAmount * 0.033
* netAmount = grossAmount - withholdingAmount

2. VAT_INVOICE

* vatAmount 계산 가능 구조로 준비
* gross/net 정의가 현재 시스템 금액 구조와 맞도록 일관성 유지
* 필요 시 initial version은 review 처리 가능

3. OVERSEAS_REMITTANCE

* 자동 실지급 계산은 하되
* 수수료/환율 이슈 때문에 기본은 review 권장

모든 계산은 소수점/반올림 기준을 명확히 통일할 것.

---

8. 상태 자동 분류

정산 상태 예시:

* draft
* ready
* pending_review
* paid

자동 생성 시 분류 규칙:

ready:

* 필수 정보 완비
* 국내 지급
* 3.3% 또는 명확한 세금 유형
* 계좌 정보 정상
* 예외 없음

pending_review:

* 해외송금
* 계좌 예외
* 세금 유형 불명확
* 지급 명의 불일치 가능성
* 수동 조정 필요
* VAT/위수탁 등 복잡 케이스

draft:

* 정보 부족으로 계산 미완료 시만 사용
* 가능하면 ready / pending_review 중심으로 간다

---

9. 예외 처리 규칙 (중요)

아래 케이스는 무조건 pending_review 로 보낸다.

* 지급 계좌 정보 없음
* 통번역사 이름과 예금주 불일치 플래그 존재
* 해외 송금 대상
* 세금 유형 불명확
* 금액 0 또는 음수
* 이미 지급완료된 task와 충돌
* 관리자 수동 수정 이력 존재

reviewReason 필드에 사유를 명시:
예:

* 해외 송금 대상
* 지급 계좌 확인 필요
* 세금 유형 검토 필요
* 지급 정보 불완전

---

10. 정산 목록/화면 반영

정산 화면에서 아래가 보여야 한다.

추가 표시 항목:

* 지급 예정일
* 정산 유형
* 원천세/부가세 여부
* 실지급액
* 자동 생성 여부
* 검토 필요 사유

상단 요약:

* 오늘 지급 예정
* 지급 대기
* 검토 필요
* 지급 완료

정렬 추천:

* 지급 예정일 빠른순
* 검토 필요 우선
* ready 우선

---

11. 대시보드 연동

대시보드 운영 알림과 연결한다.

예:

* 지급 예정인데 아직 ready/paid 처리 안 된 건
* pending_review 건수
* 오늘 지급 대상 건수

즉,
정산 자동화 결과가 대시보드 KPI/알림에도 반영되어야 한다.

---

12. 로그 기록

자동 생성/변경 이력을 logs에 남긴다.

예:

* settlement_auto_created
* settlement_auto_classified
* settlement_marked_ready
* settlement_marked_review

로그 내용:

* settlementId
* taskId
* oldStatus
* newStatus
* reason

---

13. 관리자 액션

관리자는 아래만 하면 되게 만든다.

ready 건:

* 지급 처리

pending_review 건:

* 검토 후 ready 전환
* 필요 시 수동 수정

paid 처리 시:

* paidAt 기록
* paymentMethod 기록
* memo 기록 가능

즉,
사람은 계산이 아니라 승인/예외처리에 집중하게 한다.
