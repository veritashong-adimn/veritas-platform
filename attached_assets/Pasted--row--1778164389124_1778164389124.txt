지시문

견적/결제 구조를 실제 통번역 실무 운영 기준으로 재설계한다.

현재 문제:

* 견적 입력 구조가 아직 “번역형 공통 row 구조” 중심으로 되어 있음
* 통역인데 페이지/어절 선택이 나타남
* 통역장비인데 EN→KO 언어 표시가 나타남
* 발행예정일/입금예정일 위치가 부적절함
* 상품 구조와 견적 입력 구조가 동기화되지 않음

목표:
상품 구조와 견적 구조를 완전히 연결하고,
서비스 유형별로 견적 입력 row 구조를 분리하여
실제 통번역 운영 방식에 맞게 재설계.

   ━━━━━━━━━━━━━━━━━━

유지:

* 견적유형
* 청구방식
* 문서유형
* 발행유형
* 견적유효기간

2. 견적유효기간 기본값 변경
━━━━━━━━━━━━━━━━━━

현재:
수동 선택 중심

변경:
기본값 자동 계산.

기준:
발행일 + 30일

예:
2026-05-07 발행
→ 기본 만료일 2026-06-06 자동 입력

수정 가능 유지.

3. 발행예정일 제거
━━━━━━━━━━━━━━━━━━

견적 단계에서 제거.

이유:
세금계산/청구 단계 정보이므로 견적 단계와 분리.

━━━━━━━━━━━━━━━━━━
4. 입금예정일 이동

견적 생성 영역에서 제거.

이동 위치:

이유:
실제 프로젝트 결제관리 정보이기 때문.

━━━━━━━━━━━━━━━━━━

예:

* 장비인데 EN→KO 표시

반드시 유형별 row 구조로 분리.

━━━━━━━━━━━━━━━━━━

번역 상품 선택 시:

필드:

* 도착언어
* 세부유형
* 단위(어절/단어/페이지)
* 수량
* 단가

표시 예:
한국어 → 영어
전문번역
50페이지

7. 통역 row 구조
━━━━━━━━━━━━━━━━━━

통역 상품 선택 시:

필드:

* 언어 A
* 언어 B
* 통역방향
* 통역방식
* 시간단위(1시간/반일/종일)
* 시간수량
* 단가

양방향

표시:
한국어 ↔ 영어

예:
한↔영 동시통역
영↔일 순차통역

현재처럼:
EN→KO 구조 사용 금지.

10. 상품 구조와 견적 구조 동기화
━━━━━━━━━━━━━━━━━━

반드시:

예:

* 통역방향
* 장비단위
* 사용기간

등이 상품 기준 그대로 견적 row에 반영.

14. E2E 최소 테스트 원칙 적용
━━━━━━━━━━━━━━━━━━

이번 작업은 구조 변경이 크므로:
과도한 전체 E2E 반복 금지.

원칙:

* 관련 파일만 최소 탐색
* data-testid / aria-label 먼저 적용
* 핵심 시나리오만 테스트
* 동일 원인 2회 실패 시 즉시 중단 후 보고
* 전체 로그인→전체 플로우 반복 금지
