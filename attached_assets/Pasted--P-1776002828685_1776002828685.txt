현재 사용자관리 구조를 내부 운영 기준에 맞게 개편해라.

[목표]
지금 사용자 역할이 `고객`, `통번역사` 중심으로만 되어 있어서, 실제 우리 회사 내부 직원(관리자, PM, 운영, 영업, 정산 등)을 관리할 수 없다.
이 플랫폼은 내부 직원 + 고객 + 통번역사 + 관리자 구조를 모두 지원해야 한다.
따라서 사용자관리, 권한관리, DB, API, UI를 아래 기준으로 정리해라.

---

1. 역할 구조 재설계

---

현재 역할 구조를 점검하고, 사용자 유형(Role Type)을 아래 4개로 확장해라.

* admin (관리자)
* staff (내부직원)
* client (고객)
* linguist (통번역사)

중요:

* 지금 화면에서 보이는 `고객`, `통번역사`만 있는 드롭다운 구조를 바꿔야 한다.
* 관리자 계정도 사용자관리 목록에서 자연스럽게 관리되어야 한다.
* “직원”은 별도 개념으로 반드시 추가한다.
* 기존 데이터와 최대한 호환되도록 마이그레이션 전략도 같이 반영해라.

---

2. DB 구조 수정

---

현재 users 테이블 및 관련 스키마를 점검하고 아래 필드를 반영해라.

필수 반영:

* role_type: enum 또는 text
  허용값:

  * admin
  * staff
  * client
  * linguist

내부 직원 관리를 위한 필드 추가:

* company_id nullable
* department nullable
* job_title nullable

설명:

* admin/staff는 기본적으로 내부 사용자다.
* client는 고객 계정이다.
* linguist는 통번역사 계정이다.
* company_id는 우선 nullable로 두되, 향후 멀티 조직 확장 가능하게 설계한다.
* 당장은 우리 회사 1개 조직 기준으로 써도 되지만 구조는 확장 가능하게 유지한다.

기존 데이터 처리:

* 기존 `관리자` 계정은 role_type=admin 으로 정리
* 기존 `고객` 계정은 role_type=client
* 기존 `통번역사` 계정은 role_type=linguist
* 내부 운영용 계정이 있다면 role_type=staff 또는 admin 으로 정리 가능하게 한다

주의:

* 기존 로그인/인증 로직이 깨지지 않도록 한다.
* 기존 JWT / auth middleware / 권한 체크 코드도 role_type 기준으로 정리한다.

---

3. RBAC 구조 정리

---

현재 “역할”과 “시스템 역할(RBAC)”이 혼재되어 있으면 분리 정리해라.

개념 분리:

1. 역할 유형(role_type)

* admin
* staff
* client
* linguist

2. 시스템 권한(permission or rbac_role)

* 전체권한
* 프로젝트관리
* 배정관리
* 결제관리
* 정산관리
* 고객관리
* 읽기전용
  등으로 확장 가능하게

중요:

* 역할 유형과 세부 권한은 다른 개념이다.
* 예를 들어:

  * admin: 전체권한 가능
  * staff: PM / 운영 / 정산 등 세부 권한 배정 가능
  * client: 제한된 고객 포털 권한
  * linguist: 자기 작업/정산 조회 권한

현재 UI에 있는 “시스템 역할(RBAC)” 필드는 유지하되,
오직 admin/staff 에게만 의미 있게 적용되도록 정리해라.

client / linguist 에게는 불필요하면 숨기거나 비활성화해도 된다.

4. 사용자관리 UI 수정

---

현재 사용자관리 화면을 점검하고 아래처럼 수정해라.

[역할 컬럼 / 역할 변경 드롭다운]
기존:

* 고객
* 통번역사

수정 후:

* 관리자
* 직원
* 고객
* 통번역사

변경사항:

* 목록의 역할 표시도 위 기준으로 통일
* 역할 변경 드롭다운도 위 4개 항목 지원
* 관리자 계정은 변경 제한 정책이 필요하면 유지
* 자기 자신을 실수로 강등하거나 비활성화하지 못하도록 안전장치 추가

추가 표시 필드:

* 부서
* 직책
* 소속 회사(있으면)

표시 원칙:

* admin/staff 에게만 부서/직책 표시
* client 는 고객 계정으로 표시
* linguist 는 통번역사 계정으로 표시

---

5. 사용자 생성/수정 흐름 정리

---

사용자 생성 흐름이 뒤섞여 있다면 아래처럼 개념을 분리해라.

A. 내부 사용자 생성

* 관리자만 생성 가능
* role_type: admin 또는 staff
* 이메일, 이름, 비밀번호(또는 초기 비밀번호), 부서, 직책 입력 가능

B. 고객 계정 생성

* 고객용 계정
* 필요시 거래처/담당자/고객관리 데이터와 연결 가능하게 구조 고려

C. 통번역사 계정 생성

* 통번역사 프로필과 연결 가능해야 함
* 기존 통번역사 리소스와의 연결 여부도 점검

중요:

* 지금 당장 생성 UI를 전부 새로 만들지 못해도,
  최소한 사용자 수정/역할 변경/표시 구조는 먼저 정상화해라.

6. API 수정

---

현재 사용자 목록 조회 / 사용자 수정 / 역할 변경 관련 API를 점검하고 수정해라.

반영할 것:

* role_type 반환
* department 반환
* job_title 반환
* company_id 반환

역할 변경 API:

* 고객 ↔ 통번역사 ↔ 직원 ↔ 관리자 변경 가능 여부 정책 명확화
* 최소한 admin/staff/client/linguist 로 저장 가능하게 수정

안전장치:

* 마지막 admin 계정을 일반 직원이나 고객으로 변경 금지
* 로그인한 자기 자신의 admin 권한 제거 금지
* 자기 자신 비활성화 금지
