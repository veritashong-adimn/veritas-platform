지시문
목표:
현재 이미 정리된 정산 리스트/자동화 구조를 유지한 상태에서,
프로젝트 단위로 판매정보 / 통번역사 수행정보 / 결제·정산 흐름을 한 화면에서 파악할 수 있는
“프로젝트 상세 컨트롤타워 화면”을 구현한다.
중요:
•	기존 정산 리스트를 다시 만드는 작업이 아니다
•	현재 정산 자동화, 컬럼 구조, 상태 분류는 유지한다
•	새 화면은 “프로젝트 상세 종합 상황판” 역할이다
핵심 방향:
이 화면은 기존 CRM 구조를 단순 개선하는 것이 아니라,
“판매 / 수행 / 결제 / 지급 흐름을 한 화면에서 판단하고 실행하는”
플랫폼 중심 UI로 새롭게 설계한다.
________________________________________
1.	이번 작업 범위
신규 또는 개선 대상:
•	프로젝트 상세 페이지(또는 상세 탭) 컨트롤타워화
•	상단 요약 카드
•	판매정보 섹션
•	통번역사 수행/정산 섹션
•	결제/입금/지급 흐름 섹션
이번 단계에서 제외:
•	정산 리스트 전체 구조 재작업
•	실제 은행 자동 송금
•	회계 외부 연동
•	과도한 손익 분석 고도화
________________________________________
2.	화면 목적
프로젝트 하나를 열었을 때 아래를 한 번에 파악 가능해야 한다.
1.	고객에게 얼마에 판매했는가
2.	누가 수행했고 언제 납품했고 얼마를 지급해야 하는가
3.	고객 결제와 통번역사 지급이 어디까지 왔는가
즉,
프로젝트 단위의 “판매 / 수행 / 돈 흐름”을 한 화면에서 보는 구조다.
________________________________________
3.	전체 화면 구조
[상단]
•	프로젝트 기본 정보 + 요약 카드
[중단 1]
•	판매정보
[중단 2]
•	통번역사 정보
[중단 3]
•	결제 / 정산 / 지급 정보
[하단]
•	메모 / 로그 / 확장 정보 (선택)
________________________________________
4.	상단 헤더 + 요약 카드
헤더 기본 정보:
•	프로젝트명
•	고객사 / 부서
•	서비스 유형
•	언어쌍
•	현재 프로젝트 상태
•	납기일 / 완료 여부
요약 카드:
•	판매금액
•	총 지급예정액
•	예상 차액(1차는 단순 계산)
•	고객 결제 상태
•	통번역사 지급 상태
주의:
예상 차액은
판매금액 - 총 실지급예정액
수준의 1차 단순 계산으로 시작한다.
너무 복잡한 회계 구조로 가지 말 것.
________________________________________
5.	판매정보 섹션
목적:
고객 기준 판매/청구 구조를 보여준다.
표시 항목:
•	견적일
•	판매 상품 / 서비스명
•	수량 / 단가
•	공급가액
•	부가세
•	총 판매금액
•	비고
중요:
기존 CRM의 판매정보를 참고하되,
현재 플랫폼 톤에 맞게 간결한 테이블 또는 카드형 리스트로 구성한다.
________________________________________
6.	통번역사 정보 섹션
목적:
누가 수행했고, 언제 납품했고, 얼마 지급해야 하는지 보여준다.
기본 컬럼:
•	통번역사
•	통번역 유형 / 언어
•	납품일
•	지급예정일
•	원금액
•	원천세
•	실지급액
•	정산유형
•	상태
표시 방식:
•	이름 메인 + 이메일 보조
•	납품일은 실제 완료일 기준 필드 우선 사용
•	현재는 settlements.createdAt fallback 허용
•	장기적으로는 task.completedAt 또는 deliveryCompletedAt 연결 구조 준비
확장 상세(접기/펼치기 또는 상세 모달):
•	교통비
•	출장비
•	이동보상
•	저작권료
•	취소보상
•	기타 추가비용
원칙:
리스트는 실행용으로 단순,
상세는 계산 검증용으로 확장한다.
________________________________________
7.	결제 / 정산 / 지급 정보 섹션
목적:
프로젝트의 돈 흐름을 보여준다.
표시 항목:
•	고객 청구일
•	고객 입금일
•	입금금액
•	미수 여부
•	통번역사 지급예정일
•	지급완료일
•	지급금액
•	지급상태
추가 요약:
•	고객 결제 완료 / 미결제
•	통번역사 지급 완료 / 미지급
•	현재 프로젝트 기준 입금-지급 흐름
핵심:
받은 돈 / 줄 돈 / 아직 안 준 돈이 한눈에 보여야 한다.
________________________________________
8.	데이터 구조 / API
가능하면 프로젝트 상세 전용 API를 만든다.
예:
GET /admin/projects/:id/control-tower
응답에 포함:
•	프로젝트 기본 정보
•	판매정보 집계
•	배정된 통번역사 목록
•	통번역사별 정산 요약
•	결제/입금 정보
•	지급 정보
•	요약 카드용 집계값
원칙:
프론트에서 여러 API를 조합하지 말고,
가능한 한 전용 상세 API 하나로 그린다.
________________________________________
9.	납품일 처리 기준
현재 완료 보고 기준으로 settlements.createdAt을 납품일처럼 사용하고 있으나,
이번 작업에서는 아래 기준으로 간다.
우선순위:
1.	task.completedAt
2.	실제 납품 완료일 필드(deliveryCompletedAt 등)
3.	없으면 settlements.createdAt fallback
즉,
현재 데이터 구조와 호환되게 하되,
장기적으로 실제 납품일 필드를 우선 사용하도록 설계한다.
________________________________________
10.	UI 원칙
반드시 지킬 것:
•	한 화면에서 전체 흐름 파악 가능
•	상단은 요약, 중단은 섹션별 상세
•	기존 CRM처럼 모든 컬럼을 한 표에 몰아넣지 말 것
•	현재 플랫폼 카드/배지 스타일 유지
•	정산 화면과 중복되는 정보는 “프로젝트 기준 문맥”으로 재배치할 것
________________________________________
11.	액션 버튼
상단 또는 섹션 내부에 배치:
•	견적 보기
•	결제 보기
•	정산 보기
•	통번역사 배정/수정
•	지급 처리 화면 이동
원칙:
핵심 액션만 상단,
세부 액션은 각 섹션 내부.
________________________________________
12.	구현 우선순위
1차:
•	프로젝트 상세 컨트롤타워 뼈대
•	상단 요약 카드
•	판매정보 섹션
•	통번역사 정보 섹션
•	결제/정산 정보 섹션
2차:
•	추가 비용 상세
•	손익 고도화
•	로그/이력
•	수정 액션 연동
________________________________________
13.	완료 기준
작업 완료 후:
•	프로젝트 하나 열면 판매/수행/결제/지급 흐름이 보일 것
•	정산 리스트로 들어가지 않아도 프로젝트 수준 판단 가능할 것
•	기존 CRM보다 가독성이 좋고, 필요한 정보는 더 구조적으로 보일 것
•	현재 구현된 정산 자동화 구조와 충돌하지 않을 것
________________________________________
14.	완료 보고 형식
•	수정/추가한 파일
•	신규 API 또는 집계 로직
•	상단 요약 카드 항목
•	판매정보 / 통번역사 정보 / 결제정산 정보 필드
•	납품일 처리 기준(task.completedAt / fallback 등)
•	테스트 결과
•	화면 캡처 또는 변경 요약
핵심 한 줄:
“정산 리스트는 실행 화면으로 유지하고,
프로젝트 상세는 판매·수행·결제·지급을 한 화면에서 보는 컨트롤타워로 만든다.”
