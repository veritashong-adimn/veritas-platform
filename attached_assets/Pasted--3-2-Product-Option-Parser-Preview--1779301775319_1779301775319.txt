상품 마스터 3-2단계 — Product + Option Parser 정교화 작업

중요:
이번 작업은
기존 Preview 분석 기능을
“더 정확하게 해석”
하도록 개선하는 단계이다.

현재 Preview 구조 자체는 성공했다.

하지만 실제 CRM 데이터에서:

- ISO 언어코드
- 일반/전문/긴급
- 산업 키워드

등을 충분히 분리하지 못하고 있다.

따라서:
이번 단계는
Parser 정교화 작업이다.

==================================================
절대 금지
==================================================

이번 단계에서도:

- 기존 products DB rewrite 금지
- 자동 상품 삭제 금지
- 자동 merge 금지
- 실제 schema 변경 금지
- quote 구조 변경 금지
- 자동견적 로직 변경 금지

즉:
이번 단계도
분석/추천/preview 강화까지만 수행.

==================================================
1. ISO Language Pair Parser 추가
==================================================

현재 CRM 데이터에는:

- ko→en
- en→ko
- ja→ko
- zh→en
- en→ms
- zh→ph

등 ISO language pair 기반 상품이 매우 많다.

현재 parser는:
영한/한영 중심으로만 일부 분석 중이다.

--------------------------------------------------

따라서:

ISO language code parser를 추가한다.

--------------------------------------------------

지원 예시:

ko
en
ja
zh
fr
de
es
ru
ar
pt
vi
th
id
ms
hi
ph
tr
it
mn

등.

--------------------------------------------------

다음 패턴 모두 인식:

- ko→en
- ko-en
- ko_en
- en→ko
- ja→ko

등.

--------------------------------------------------

normalize:

예:
ko→en

↓

Language Pair:
한국어 ↔ 영어

Direction:
ko→en

--------------------------------------------------

중요:
Preview 표시용 Language Pair와
실제 Direction은 분리.

==================================================
2. Difficulty Option 분리
==================================================

다음 키워드는:
Product가 아니다.

Option/Difficulty로 분리한다.

--------------------------------------------------

키워드:

- 일반
- 전문
- 긴급
- 특급
- VIP
- 고급
- 프리미엄

--------------------------------------------------

예:

한영 일반번역

↓

Product:
번역

Language Pair:
한국어 ↔ 영어

Direction:
ko→en

Difficulty:
일반

--------------------------------------------------

전문번역

↓

Product:
번역

Difficulty:
전문

--------------------------------------------------

중요:
“일반번역”
전체를 Product로 저장하지 않는다.

==================================================
5. Translation vs Interpretation 규칙 강화
==================================================

--------------------------------------------------
번역(TR)
--------------------------------------------------

direction 중요.

- ko→en
- en→ko

유지.

--------------------------------------------------
통역(IN)
--------------------------------------------------

기본:
bidirectional 우선 추천.

예:

영어동시통역

↓

Language Pair:
한국어 ↔ 영어

Direction:
bidirectional

--------------------------------------------------

단,
향후 글로벌 확장 고려하여:

- ko→en
- en→ko

도 parser에서 유지 가능하게 한다.

→ 동시통역 + 한국어↔영어 + bidirectional

법률/의료가 Product에 남지 않는지.

6.
기존 import workflow 정상 유지되는지.

7.
기존 DB 데이터 변경 없는지.

8.
Babel parse / Vite HMR 오류 없는지 확인.
