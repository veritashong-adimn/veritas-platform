[작업명]
담당자 등록 모달 - 브랜드/부서 선택 기능 추가

[목표]
현재 담당자 등록 화면에서는 거래처(회사)만 선택 가능하고,
해당 거래처 하위의 브랜드/부서(divisions)를 선택할 수 없어
예: 리치몬트코리아(반클리프아펠) 소속 담당자 등록이 불가능하다.

이를 개선하여
담당자 등록 시
- 거래처(회사) 선택
- 해당 거래처의 브랜드/부서 선택
이 가능하도록 수정한다.

즉,
“회사 단위 소속 + 브랜드/부서 단위 소속”을 함께 저장할 수 있게 한다.

--------------------------------------------------
1. 문제
--------------------------------------------------

현재 담당자 등록 모달은
거래처(company)만 선택 가능하고,
선택된 거래처 하위의 브랜드/부서(divisions)를 지정할 수 없다.

예:
- 거래처: 리치몬트코리아
- 브랜드/부서: 반클리프아펠

이 경우
담당자 성묘연을
“리치몬트코리아 소속이면서 반클리프아펠 담당자”
로 등록해야 하는데,
현재 UI에서는 브랜드/부서 선택이 불가능하다.

--------------------------------------------------
2. 수정 목표
--------------------------------------------------

담당자 등록 시 아래 구조를 지원할 것:

- 거래처(회사) 선택
- 브랜드/부서 선택 (선택 가능, 없으면 비워둘 수 있음)

예:
거래처: 리치몬트코리아
브랜드/부서: 반클리프아펠
담당자명: 성묘연

--------------------------------------------------
3. UI 수정
--------------------------------------------------

담당자 등록 모달에
거래처 선택 필드 바로 아래에
새 필드 추가:

필드명:
브랜드/부서

placeholder 예시:
브랜드명 또는 부서명 선택 (선택)

동작 방식:
- 거래처를 먼저 선택해야 활성화
- 거래처 미선택 시 disabled
- 거래처 선택 후 해당 거래처의 divisions 목록 로드

표시 형식 예시:
- 반클리프아펠
- 까르띠에
- 몽블랑
- 시계사업부
- 마케팅팀

--------------------------------------------------
4. 데이터 로딩 방식
--------------------------------------------------

거래처 선택 시
해당 companyId 기준으로 division 목록을 조회한다.

예상 API:
GET /api/admin/divisions?companyId={companyId}
또는 현재 divisions 조회 API가 있으면 그것을 사용

필수 동작:
- 거래처 변경 시 브랜드/부서 선택값 초기화
- 새 거래처에 해당하는 목록으로 다시 로드

--------------------------------------------------
5. 저장 구조
--------------------------------------------------

담당자 저장 시
기존 contact 생성 payload에 divisionId를 포함할 것.

예시 payload:
{
  "companyId": 123,
  "divisionId": 456,
  "name": "성묘연",
  "department": "...",
  "position": "...",
  "email": "...",
  "mobile": "...",
  "phone": "...",
  "memo": "...",
  "isPrimary": true,
  "isQuoteContact": false,
  "isBillingContact": false,
  "isActive": true
}

주의:
- divisionId는 nullable 허용
- 브랜드/부서를 선택하지 않으면 null 저장 가능

--------------------------------------------------
6. 상세/목록 표시도 함께 반영
--------------------------------------------------

담당자 목록 및 상세 화면에서
division 정보가 있으면 함께 표시할 것.

표시 예시:
리치몬트코리아 / 반클리프아펠

또는 상세에서:
거래처: 리치몬트코리아
브랜드/부서: 반클리프아펠

현재 division 정보가 저장되어도 화면에 안 보이면 의미가 없으므로
등록 + 저장 + 조회까지 연결할 것.

--------------------------------------------------
7. 수정 모드에도 동일 적용
--------------------------------------------------

담당자 상세 > 수정 모드에서도
브랜드/부서 변경 가능하게 할 것.

즉,
- 신규 등록 모달
- 담당자 수정 모드
둘 다 동일하게 division 선택 필드가 있어야 한다.

--------------------------------------------------
8. UX 규칙
--------------------------------------------------

1) 거래처 먼저 선택
2) 브랜드/부서는 해당 거래처 기준 목록만 표시
3) 브랜드/부서 없는 회사는 비워둘 수 있음
4) 거래처 변경 시 기존 브랜드 선택은 자동 초기화

--------------------------------------------------
9. 검증 규칙
--------------------------------------------------

- divisionId가 선택되면 반드시 해당 companyId 소속 division인지 검증
- 다른 회사 division을 잘못 저장하지 않도록 백엔드에서도 검증할 것

즉,
프론트에서 제한하고
백엔드에서도 companyId-divisionId 관계를 다시 확인할 것

--------------------------------------------------
10. 영향 범위
--------------------------------------------------

이번 작업 범위:

포함:
- 담당자 등록 모달
- 담당자 수정 모달
- 담당자 목록/상세 표시
- 저장 payload에 divisionId 포함

제외:
- division 생성/삭제 기능 추가
- 거래처 관리 화면 구조 변경
- 프로젝트 등록 화면 변경

--------------------------------------------------
11. 테스트
--------------------------------------------------

반드시 아래 테스트 수행:

1) 거래처 선택 전에는 브랜드/부서 필드가 비활성화되는지
2) 리치몬트코리아 선택 시 반클리프아펠 등 해당 division 목록이 뜨는지
3) 브랜드/부서 선택 후 담당자 등록이 정상 저장되는지
4) 저장 후 목록/상세에서 division 정보가 보이는지
5) 담당자 수정 시 브랜드/부서 변경이 가능한지
6) 다른 회사 division을 강제로 넣을 경우 백엔드에서 차단되는지

--------------------------------------------------
12. 완료 기준
--------------------------------------------------

다음이 모두 만족되면 완료:

- 담당자 등록 시 브랜드/부서 선택 가능
- 거래처 기준 division 목록이 올바르게 로드됨
- divisionId가 저장됨
- 담당자 목록/상세/수정에서 브랜드/부서 정보가 반영됨
- companyId와 divisionId 관계 검증이 적용됨
