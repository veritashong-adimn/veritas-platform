견적 수정/재생성 form 내
“정식 상품 등록 요청” 구조를
상품 유형별 conditional form 구조로 개선한다.

대상:
- artifacts/web-app/src/components/admin/ProjectDetailModal.tsx
- 커스텀 상품 등록 요청 form 영역

목표:
현재 정식 상품 등록 요청 form은
번역/통역/장비/실비가
하나의 공통 form을 공유하고 있다.

그 결과:

- 장비인데 언어쌍 노출
- 실비인데 페이지 단위 노출
- 통역인데 번역 단위 노출

등,
상품 유형과 맞지 않는 필드가 같이 표시되고 있다.

이는:
- 데이터 의미 혼합
- 입력 혼란
- 상품 등록 품질 저하
- BI/AI 데이터 품질 저하

문제를 발생시킨다.

따라서:
정식 상품 등록 요청 form을

“상품 유형별 conditional form”

구조로 변경한다.

==================================================
1. 핵심 원칙
==================================================

모든 유형에 모든 필드를 보여주는 구조 금지.

선택된 상품 유형에 따라:
필요한 필드만 노출.

즉:
- 번역 = 번역용 필드
- 통역 = 통역용 필드
- 장비 = 장비용 필드
- 실비 = 실비용 필드

만 표시.

==================================================
2. 번역 요청 form
==================================================

서비스 유형:
- 번역

노출 필드:
- 언어쌍
- 단위
  - 페이지
  - 단어
  - 글자
  - 어절
- 제안 단가
- 메모

숨김:
- 사용기간
- 장비 수량 구조
- 실비 금액 전용 구조

==================================================
3. 통역 요청 form
==================================================

서비스 유형:
- 통역

노출 필드:
- 언어쌍
- 단위
  - 시간
  - 일
  - 회
- 제안 단가
- 메모

선택 가능(optional):
- 동시/순차/위스퍼링 등

숨김:
- 페이지
- 장비 사용기간
- 실비 금액 구조

==================================================
4. 장비 요청 form
==================================================

서비스 유형:
- 장비

노출 필드:
- 장비명
- 단위
  - 개
  - 세트
- 사용기간
  - 1일
  - 2일
  - 3일
- 제안 단가
- 메모

숨김:
- 언어쌍
- 페이지
- 번역 단위

==================================================
5. 실비 요청 form
==================================================

서비스 유형:
- 실비

노출 필드:
- 항목명
- 금액
- 메모

숨김:
- 언어쌍
- 페이지
- 사용기간
- 장비 단위

==================================================
6. 서비스 유형 변경 시 동작
==================================================

서비스 유형 변경 시:

- form 필드 즉시 변경
- 이전 유형 값은 초기화

예:
번역 → 장비 변경 시:
- 언어쌍 제거
- 페이지 제거
- 사용기간 구조 활성화

자연스럽게 전환되게 한다.

==================================================
7. UX 방향
==================================================

중요:
form이 더 커지거나 복잡해지면 안 된다.

즉:
- 필요한 필드만 노출
- conditional hidden 적극 사용
- compact SaaS workflow 유지
- ERP형 form 구조 금지

==================================================
8. 데이터 품질 방향
==================================================

이번 작업의 핵심은:
“UI 예쁘게 만들기”
가 아니라,

상품 유형별 데이터 의미를 유지하는 것이다.

즉:
- 데이터 구조 명확화
- 상품 등록 품질 향상
- 관리자 승인 효율 향상
- BI 품질 향상
- AI 추천 정확도 향상

을 목표로 한다.

==================================================
주의사항
==================================================

- 기존 커스텀 상품 저장 구조 유지
- 기존 견적 workflow 깨지지 않게 유지
- DB/API/document service 수정은 최소 범위만 진행
- 관련 없는 버그는 수정하지 말고 보고만 한다.
- vertical spacing 과도 증가 금지
- 과도한 카드화 금지
- ERP/table/grid 느낌 강화 금지
- opacity/transform/blur animation 추가 금지

==================================================
완료 후 확인
==================================================

1. 상품 유형별로 필요한 필드만 노출되는지 확인한다.
2. 장비 선택 시 언어쌍/페이지 등이 숨겨지는지 확인한다.
3. 실비 선택 시 금액 중심 구조로 변경되는지 확인한다.
4. 번역/통역 선택 시 언어쌍 구조가 정상 노출되는지 확인한다.
5. 서비스 유형 변경 시 이전 값이 안전하게 초기화되는지 확인한다.
6. compact workflow가 유지되는지 확인한다.
7. Babel parse / Vite HMR 오류 없는지 확인한다.
