작업을 시작하기 전에 챗지피티가 정리한 내용을 너에게 사전설명서 형식으로 먼저 보여줄거야, 내용을 미리 인지하기 바란다. 작업 시작은 사전 설명서 이후에 지시문을 올리주면 시작하도록 하자.

사전 설명서
현재 진행하려는 작업은
“플랫폼 상품 구조 전체를 한 번에 뒤엎는 작업”
이 아니다.

핵심 목표는:

“현재 베리타스의 실제 운영 구조에 맞게
플랫폼 상품 마스터 DB를 안정적으로 정리하는 것”

이다.

==================================================
현재 상황
==================================================

기존 CRM에서 사용하던 상품 데이터를
플랫폼 상품 DB로 이관하려고 한다.

현재 데이터는:

- 번역
- 통역
- 통역장비
- 잡(실비/부대비용)

등으로 나뉘어 있으며,
상품명 안에:

- 언어쌍
- 일반/전문
- 산업 의미
- 통역 유형
- 장비 유형

등의 의미가 섞여 있다.

예:
- 영어 전문번역
- 영어-태국어 동시통역
- 동시통역장비(FM)
- 퀵비용

등.

==================================================
매우 중요한 플랫폼 방향
==================================================

현재 논의 결과,
상품이 모든 의미를 담당하는 구조는
현실 운영과 맞지 않는다고 판단했다.

왜냐하면 실제 운영에서는:

- 대부분 전문번역
- 일반/전문 경계 모호
- 프로젝트마다 난이도 다름
- 산업마다 요구 수준 다름
- 긴급도 중요
- 고객별 품질 기준 다름

이기 때문이다.

따라서 플랫폼 방향은:

상품
=
기본 서비스 유형

프로젝트
=
실제 난이도/산업/긴급도/보안등급

으로 역할을 분리하는 방향이다.

즉:
상품 taxonomy는 단순화하고,
실제 업무 의미는 프로젝트 구조에서 처리하는 방향이다.

==================================================
중요:
이번 작업 범위
==================================================

이번 단계에서는:

“상품 마스터 taxonomy 정리”

까지만 진행한다.

즉:
- 상품 구조 정리
- 상품 유형 정리
- 실비 분리
- 상품코드 정리
- import 구조 안정화

까지만 한다.

--------------------------------------------------

아직 건드리지 않는 것:

- 프로젝트 난이도 구조
- 자동 견적 로직
- 번역사 매칭 구조
- AI 추천 구조
- quote item 구조 대규모 변경
- 프로젝트 workflow 변경

이것들은 이후 단계에서 진행한다.

==================================================
현재 상품 taxonomy 방향
==================================================

1. 번역
- 번역
- 감수
- 원어민감수
- 공증번역

--------------------------------------------------

2. 통역
- 통역
- 수행통역
- 화상통역
- 전화통역

※ 동시/순차 등은
향후 프로젝트/서비스 속성으로 분리 가능.

--------------------------------------------------

3. 장비
- FM 장비
- 적외선 장비
- 부스
- 리시버

--------------------------------------------------

4. 실비
- 배송/퀵
- 교통비
- 출장비
- 식비
- 숙박
- 기타 실비

==================================================
중요한 방향
==================================================

현재:
- 퀵비용 등이 번역 상품 안에 섞일 수 있음.

반드시:
실비 taxonomy로 분리한다.

==================================================
상품코드 방향
==================================================

현재 임시 import 코드:
- TR-0001
- IN-0001

등은 임시 구조로 사용 가능.

다만:
언어/산업/난이도까지
과도하게 상품코드에 넣지 않는다.

추천 방향:
- TR = 번역
- IN = 통역
- EQ = 장비
- EX = 실비

정도의 가벼운 taxonomy 기반 유지.

==================================================
작업 진행 방식
==================================================

이번 작업은
한 번에 전체 구조를 변경하지 않는다.

반드시 단계별 진행:

1단계:
상품 taxonomy 정리

2단계:
상품 import 구조 안정화

3단계:
커스텀 상품 → 정식 상품 요청 구조

4단계:
프로젝트 의미 구조 분리

순서로 진행한다.

==================================================
매우 중요
==================================================

이번 작업은:

“ERP형 상품 시스템”

을 만드는 것이 아니다.

핵심은:

“실제 통번역 운영 workflow에 맞는
유연한 SaaS 상품 구조”

를 만드는 것이다.

따라서:
- 과도한 상품 세분화 금지
- 상품에 모든 의미를 넣는 구조 금지
- 지나친 코드 체계 금지
- 복잡한 grid/table 구조 금지

방향 유지.
