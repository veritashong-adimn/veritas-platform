중요:
현재 Product + Option 구조 방향은 맞다.

이번 단계는:
구조 변경이 아니라

- parser 정밀도 향상
- 운영 UX polishing
- Product 후보 안정화

작업이다.

즉:
운영자가 Preview 결과를
“신뢰할 수 있는 수준”
으로 다듬는 단계이다.

==================================================
절대 금지
==================================================

이번 단계에서도:

- 기존 products DB rewrite 금지
- 자동 상품 삭제 금지
- 자동 merge 금지
- quote 구조 변경 금지
- 자동견적 로직 변경 금지
- schema 변경 금지

즉:
분석/preview 품질 개선만 수행.

==================================================
1. Unknown ISO Language Code 처리
==================================================

현재 문제:

예:
ky→ko 일반번역

↓

Product 후보:
ky→ko번역

처럼
미지원 ISO code가 Product에 남는다.

--------------------------------------------------

수정 방향:

지원하지 않는 언어코드라도:

xx→yy
xx-yy
xx_yy

패턴이면:
우선 language token으로 제거한다.

--------------------------------------------------

예:

ky→ko 일반번역

↓

Product:
번역

Language Pair:
미지원언어 ↔ 한국어

Direction:
ky→ko

--------------------------------------------------

중요:
미지원 language code는
Product에 절대 남기지 않는다.

--------------------------------------------------

미지원 언어는:

- 회색 badge
- "미지원 언어코드"

표시 가능.

==================================================
2. Product 후보 안정화
==================================================

현재 문제:

Product 후보에:

- 번역 옵션
- ky→ko번역
- en→sr번역

등이 남는다.

--------------------------------------------------

수정 방향:

Product 후보는 항상:
실제 서비스 유형만 표시.

--------------------------------------------------

허용 예시:

- 번역
- 감수
- 원어민감수
- 공증번역
- 동시통역
- 순차통역
- 수행통역
- 화상통역
- 전화통역

--------------------------------------------------

금지:

- 번역 옵션
- 법률번역
- 일반번역
- ko→en번역

등.

--------------------------------------------------

“옵션화 가능”
은 별도 badge/chip 으로만 표시.

예:

번역 [옵션화 가능]

==================================================
3. Parsing Priority 정리
==================================================

현재 parser는
패턴 위주 처리라
순서 안정성이 부족하다.

--------------------------------------------------

반드시 다음 순서로 parsing:

1.
언어 token 제거

2.
난이도 제거

3.
산업 제거

4.
남은 핵심 service keyword 추출

--------------------------------------------------

예:

영한 긴급 법률번역

↓

1.
영한 제거

2.
긴급 제거

3.
법률 제거

4.
남은:
번역

==================================================
4. Language Pair / Direction 철학 통일
==================================================

현재 문제:

언어쌍 칸에:
영어 → 말레이어

처럼 direction 개념이 섞여 있음.

--------------------------------------------------

플랫폼 철학 통일:

Language Pair:
관계형

Direction:
방향형

--------------------------------------------------

예:

Language Pair:
영어 ↔ 말레이어

Direction:
en→ms

--------------------------------------------------

통역(IN):

기본:
bidirectional 우선.

예:

영어동시통역

↓

Language Pair:
한국어 ↔ 영어

Direction:
↔ 양방향

등.

--------------------------------------------------

예:

↓

Product:
원어민감수

벵골어

6. 난이도/산업 badge UX 정리
==================================================

현재 badge 수가 많아지면서
ERP 느낌이 올라오기 시작함.

--------------------------------------------------

수정 방향:

- 작은 chip 유지
- 연한 색상 유지
- compact spacing 유지
- 과한 강조 금지

--------------------------------------------------

우선순위:

Product
>
Language Pair
>
Direction
>
Industry
>
Difficulty

순으로 시각 강조.

==================================================
7. 빈 Option UX 정리
==================================================

현재:
산업/난이도 없는 경우
“—”
표시가 많다.

--------------------------------------------------

수정 방향:

값 없으면:
- 흐린 empty 상태
- muted text
- 또는 빈칸

정도로 정리.

ERP식 과한 placeholder 금지.

--------------------------------------------------

--------------------------------------------------

[산업 옵션형]

처럼 badge 분리.

10. 최종 확인
==================================================

반드시 확인:

1.
ky→ko 일반번역
→ Product:
번역
만 남는지.

2.
번역 옵션
같은 Product 후보 제거됐는지.

3.
영어 ↔ 말레이어
+
Direction en→ms
구조로 통일됐는지.

4.
벵골어원어민감수
→ 원어민감수 + 벵골어
분리되는지.

5.
난이도/산업 badge가 과하지 않은지.

6.
Preview UX가 여전히 compact한지.

7.
기존 import workflow 정상 유지되는지.

8.
기존 DB 데이터 변경 없는지.

9.
Babel parse / Vite HMR 오류 없는지 확인.
