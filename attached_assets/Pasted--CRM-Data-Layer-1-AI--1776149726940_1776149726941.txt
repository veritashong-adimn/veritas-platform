레플릿 지시문
현재 운영 중인 통번역 CRM/플랫폼에 데이터 축적(Data Layer) 1차 버전을 추가하라.
이번 단계의 목표는 데이터 상품화가 아니라, 앞으로 데이터/AI 사업으로 확장할 수 있도록 번역 데이터를 구조화하여 쌓는 것이다.
기존 프로젝트/견적/매칭/정산 흐름은 유지하고, 기존 기능을 깨지 말아라.
이번 작업은 반드시 **기존 운영 로직과 분리된 “데이터 레이어”**로 추가하라.
________________________________________
1. 구현 목표
현재 구조는 대체로 아래와 같다.
•	CRM 
•	프로젝트 생성 
•	견적 
•	번역사 매칭 
•	작업 진행 
•	완료/정산 
여기에 아래 데이터 레이어를 추가하라.
•	번역 원문/번역문을 문장 또는 세그먼트 단위로 저장 
•	프로젝트 단위가 아니라 데이터 단위로 축적 
•	최소한의 태깅 추가 
•	관리자 화면에서 조회 가능 
•	아직 외부 판매/내보내기 UI는 만들지 말 것 
•	이번 단계에서는 축적 + 정제 준비 + 조회까지만 구현

2. 신규 테이블 추가
Drizzle schema에 아래 신규 테이블을 추가하라.
2-1. translation_units
번역 데이터의 최소 저장 단위 테이블
필드 예시는 아래 기준으로 설계하라.
•	id 
•	projectId (FK, projects.id) 
•	taskId (nullable, tasks.id) 
•	sourceText (text, not null) 
•	targetText (text, not null) 
•	sourceLang (varchar, not null) 
•	targetLang (varchar, not null) 
•	domain (varchar, nullable) 
o	예: general / legal / finance / medical / it / marketing / technical / other 
•	translatorId (nullable, translators.id 또는 현재 번역사 테이블 FK 구조에 맞춤) 
•	qualityLevel (varchar, nullable) 
o	예: A / B / C / unknown 
•	securityLevel (varchar, not null, default 'restricted') 
o	값 예: public / internal / restricted / confidential 
•	isAnonymized (boolean, default false) 
•	anonymizedSourceText (text, nullable) 
•	anonymizedTargetText (text, nullable) 
•	sourceCharCount (integer, default 0) 
•	targetCharCount (integer, default 0) 
•	sourceWordCount (integer, default 0) 
•	targetWordCount (integer, default 0) 
•	segmentIndex (integer, not null, default 0) 
•	status (varchar, not null, default 'active') 
o	예: active / excluded / flagged 
•	createdAt 
•	updatedAt 
인덱스도 추가하라.
•	projectId 
•	translatorId 
•	domain 
•	sourceLang + targetLang 
•	securityLevel 
•	status

3. 기존 프로젝트와 연결 방식
이번 단계에서는 기존 프로젝트 완료 흐름을 깨지 말고, 아래 원칙으로 연결하라.
원칙
1.	프로젝트 자체는 기존처럼 관리 
2.	번역 결과물 또는 입력된 번역 텍스트가 있으면 
3.	프로젝트 완료 시점 또는 관리자가 수동 실행할 때 
4.	해당 번역을 세그먼트 단위로 분해해서 translation_units에 저장 
즉, 기존 운영 테이블은 그대로 두고
translation_units는 데이터 자산화용 파생 테이블로 만들어라.
________________________________________
4. 세그먼트 저장 방식
이번 1차 버전에서는 복잡한 NLP 분절까지 하지 말고, 아래처럼 단순 구현하라.
1차 분절 규칙
•	줄바꿈 단위 우선 분리 
•	줄바꿈이 없으면 문장부호 기준 분리 
•	source/target pair 수가 맞지 않으면 전체를 1개 unit로 저장 
•	빈 줄은 제거 
•	segmentIndex 순서 저장 
중요
완벽한 문장 분할보다 데이터를 안정적으로 쌓는 것이 우선이다.
실패 시 전체 텍스트를 1개 unit로 저장하는 fallback을 넣어라.
________________________________________
5. 데이터 저장 서비스 추가
백엔드에 별도 서비스 모듈을 추가하라.
예시 파일명:
•	services/translationUnitService.ts 
이 서비스에 최소 아래 함수들을 구현하라.
함수 예시
1.	segmentTranslationPair(sourceText, targetText) 
•	source/target을 세그먼트 배열로 변환 
2.	buildTranslationUnitsFromProject(projectId) 
•	프로젝트 관련 데이터 읽기 
•	번역 원문/번역문 확보 
•	세그먼트 분리 
•	translation_units insert 
3.	anonymizeTranslationUnit(unitId) 
•	단순 비식별화 처리 
•	사람 이름/회사명/이메일/전화번호 패턴을 마스킹 
•	예: [PERSON], [COMPANY], [EMAIL], [PHONE] 
4.	bulkAnonymizeByProject(projectId) 
5.	excludeTranslationUnit(unitId) 
•	status = excluded 처리 
주의
현재 단계에서는 AI 기반 익명화까지 하지 말고
정규식/룰 기반 마스킹으로만 구현하라.
________________________________________
6. 프로젝트 완료 시 자동 축적
기존 프로젝트 완료 처리 로직에 아래를 추가하라.
•	프로젝트가 완료 처리될 때 
•	sourceText/targetText 또는 번역 결과물이 존재하면 
•	translation_units 자동 생성 시도 
•	실패해도 프로젝트 완료 자체는 막지 말 것 
•	실패 시 로그만 남길 것 
즉:
•	운영 로직 우선 
•	데이터 축적 실패는 비치명적 처리

10. 데이터 기본 정책
이번 1차에서는 아래 정책으로 시작하라.
기본값
•	securityLevel 기본값: restricted 
•	qualityLevel 기본값: unknown 
•	domain 기본값: 프로젝트 카테고리/상품/메모에서 유추 가능하면 자동 지정, 아니면 general 
•	isAnonymized 기본값: false 
주의
민감 고객 데이터가 많으므로
기본은 “활용 가능 데이터”가 아니라 **“제한 데이터”**로 저장하라.

14. 완료 기준
아래 조건을 만족하면 완료로 본다.
1.	신규 schema가 정상 반영된다. 
2.	프로젝트 완료 시 translation_units가 자동 생성된다. 
3.	관리자에서 목록/상세 조회가 된다. 
4.	익명화 버튼이 동작한다. 
5.	제외 처리 가능하다. 
6.	기존 프로젝트/견적/정산/매칭 기능이 깨지지 않는다.

추가 지시
기존 코드베이스를 먼저 검토해서
현재 프로젝트/작업/번역 결과가 어디에 저장되는지 정확히 파악한 후 연결하라.
즉, sourceText/targetText 필드가 이미 있으면 그것을 재사용하고,
없으면 현재 구조에 맞게 최소 침습 방식으로 연결하라.
마이그레이션/스키마 반영 후, 샘플 프로젝트 1건으로 실제 저장까지 테스트하라.
________________________________________
중요: 이번 작업은 “번역 기능 추가”가 아니라 “데이터 자산화 기반 구축”이다. 기존 운영 기능보다 뒤에서 조용히 동작하는 구조로 구현하라.
