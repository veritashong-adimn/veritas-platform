지시문 — Review Persistence 초기 구조 구축

현재 상품 마스터 Import Preview는 다음 단계까지 완료된 상태입니다.

- Product / Option 구조 정리
- Import Preview 기반 검토 구조
- Sticky Header
- 등록 안전장치
- Confidence Score
- Review Reason
- Canonical Stabilization
- Review Workflow 초기 구조
- 승인 / 제외 / 보류 상태
- Manual Override
- Reject Reason
- 수정됨 필터

현재 상태는:
“운영자가 Preview 내부에서 판단”
까지 가능한 상태입니다.

이번 작업 목표는:

“운영자의 판단 결과를 저장 가능한 구조로 전환”

하는 것입니다.

중요:
- AI 자동학습 구현 금지
- 자동등록 엔진 구현 금지
- Confidence 자동 재계산 금지
- 기존 Import Preview UX 유지
- Sticky Header 유지
- 기존 등록 안전장치 유지
- Product/Option 철학 유지

이번 단계는:
Review Persistence(검토 이력 저장) 초기 구조만 구현합니다.

핵심 목표

현재:
운영자의 판단이 Preview state 안에서만 존재

다음:
운영자의 판단이:
- 저장 가능하고
- 다시 불러올 수 있고
- 누적 가능한 구조

로 변경.

즉:
“운영자의 판단 데이터가 자산으로 남기 시작하는 단계”

입니다.

수정 요청

1. Review Log 저장 구조 추가

운영자가 다음 행동을 한 경우:
- 승인
- 제외
- override 수정

review log 형태로 저장 가능하게 해 주세요.

최소 구조 예시:

review_logs
- rowNum
- originalText
- originalCandidate
- overriddenCandidate
- finalCandidate
- reviewStatus
- rejectReason
- confidenceScore
- reviewReasons
- reviewedAt
- reviewedBy (placeholder 가능)

중요:
초기 단계에서는 간단한 local persistence 또는 lightweight 저장 구조여도 괜찮습니다.

핵심은:
“운영자의 판단 결과가 사라지지 않는 구조”
를 만드는 것.

2. Review Session 개념 추가

한 번의 Import Preview를:
하나의 review session처럼 관리할 수 있게 준비해 주세요.

예:
review_session
- sessionId
- uploadedAt
- totalRows
- approvedCount
- rejectedCount
- modifiedCount

등.

중요:
이번 단계에서는 최소 구조만.

3. Preview 새로고침 후 상태 유지

현재:
새로고침/재진입 시:
- 승인
- 제외
- override
- rejectReason

등이 사라질 가능성이 있습니다.

이번 작업 후:
동일 session 기준으로 다시 들어와도:

- reviewStatus
- overriddenCandidate
- rejectReason

등이 유지되도록 해 주세요.

4. Review History 표시 강화

현재 override는 tooltip 수준입니다.

추가:
행 내부 또는 compact UI로 다음 표시 가능하게 해 주세요.

예:
AI 추천:
영문화

운영자 수정:
번역

즉:
“AI 추천값 vs 최종 운영자 판단”
이 구분되게.

단:
기존 compact UX 유지.

5. reviewedAt / reviewedBy 구조 준비

현재 로그인 유저 연동까지는 필요 없음.

하지만 구조상:
- reviewedAt
- reviewedBy

필드가 유지되도록 준비.

reviewedBy는 임시 placeholder 가능.

예:
"admin"
"operator"

등.

6. Reject Pattern 저장 준비

다음 정보도 함께 저장 가능하게 준비해 주세요.

- rejectReason
- reviewReasons
- originalCandidate

즉:
나중에:
“어떤 패턴이 자주 reject 되는가”
분석 가능한 구조.

중요:
이번 단계에서는 단순 저장만.
자동 분석/AI 학습 구현 금지.

7. 기존 등록 구조 유지

다음은 절대 깨지면 안 됩니다.

- Sticky Header
- Import Preview flow
- 등록 안전장치
- 승인/보류/제외 구조
- 기존 필터/정렬
- Product/Option 철학
- Canonical 구조

8. 이번 단계에서 하지 않는 것

중요:
아직 구현 금지.

- AI 자동학습
- confidence 자동 변경
- 자동 canonical 생성
- 승인 패턴 기반 AI 수정
- 자동등록 엔진

현재는:
“운영 판단 저장 단계”
만 구현.

9. 검증

수정 후 다음 확인해 주세요.

- Babel/TypeScript 통과
- 기존 Preview 렌더링 정상
- Sticky Header 유지
- 기존 등록 안전장치 유지
- 승인/제외/override 상태 유지 확인
- 새로고침 후 review 상태 유지 확인
- Review Session 생성 정상
- review log 저장 정상
- 기존 필터/정렬과 충돌 없음
- HMR 오류 없음

7. 검증 결과
