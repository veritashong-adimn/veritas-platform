[목표]
기존 "고객 등록/수정" 화면과 별도로,
B2B 실무에 맞는 "거래처 담당자 등록/수정" 기능을 신규 구축합니다.

이 기능은 거래처(회사)와 담당자(사람)를 분리 관리하기 위한 것으로,
하나의 거래처에 여러 명의 담당자를 등록하고,
프로젝트/견적/청구 시 적절한 담당자를 선택할 수 있도록 하는 것이 목적입니다.

==================================================
1. 핵심 원칙
==================================================

- 거래처(회사)와 담당자(개인)는 별도 엔티티로 분리합니다.
- 담당자는 반드시 특정 거래처에 소속되어야 합니다.
- 한 거래처에는 담당자 여러 명이 등록될 수 있습니다.
- 담당자마다 역할(기본 담당자 / 견적 담당자 / 청구 담당자)을 지정할 수 있게 합니다.
- 기존 "고객 등록/수정" 화면은 거래처 담당자 등록 화면으로 사용하지 않습니다.
- 별도의 "거래처 담당자 등록/수정" 페이지와 목록/탭을 추가합니다.

==================================================
2. 첨부 화면 기준 제거/비적용 항목
==================================================

기존 고객 등록/수정 화면에 있던 아래 항목들은
거래처 담당자 등록 화면에서는 제거합니다.

제거 대상:
- 고객구분
- 프로젝트구분
- 정보활용동의
- 성별
- 그룹명
- 주소
- 접촉거부정보
- 오류정보
- 직장팩스
- 고객등록일(별도 입력필드 제거, created_at 자동처리)
- 관리자(담당자 등록 폼에는 직접 노출하지 않아도 됨. 필요 시 내부값 처리)

위 항목들은 거래처 담당자 관리 목적과 맞지 않으므로
신규 담당자 화면에서는 사용하지 않습니다.

==================================================
3. 거래처 담당자 등록/수정 화면 필수 항목
==================================================

[기본정보]
- 거래처 (company_id) : 필수, 검색 선택 방식
- 담당자명 (name) : 필수
- 휴대폰 (mobile) : 필수 권장
- 이메일 (email) : 필수 권장

[상세정보]
- 부서 (department)
- 직책 (position)
- 직장전화 (office_phone)
- 메모 (memo)

[역할/옵션]
- 기본 담당자 여부 (is_primary)
- 견적 담당자 여부 (is_quote_contact)
- 청구 담당자 여부 (is_billing_contact)

추가 선택 항목이 필요하면 아래만 허용:
- 재직 여부/활성 여부 (is_active) ← 필요 시만 추가
기본값 true

==================================================
4. DB 스키마 추가
==================================================

company_contacts (또는 contacts) 테이블을 신규 생성해주세요.

권장 컬럼:
- id
- company_id (FK -> companies/customers/거래처 테이블)
- name varchar/text not null
- mobile varchar/text
- email varchar/text
- department varchar/text
- position varchar/text
- office_phone varchar/text
- memo text
- is_primary boolean default false
- is_quote_contact boolean default false
- is_billing_contact boolean default false
- is_active boolean default true
- created_at timestamp
- updated_at timestamp

[제약/권장사항]
1. company_id 는 필수
2. 거래처 삭제 시 담당자 처리 정책 확인
   - soft delete 구조가 없으면 우선 연결 유지
3. 이메일 중복은 전체 시스템에서 완전 유니크로 막지 말고,
   같은 거래처 내 중복 여부만 검토 가능하도록 합니다.
   (실무상 대표메일 공유 가능성 있음)
4. 휴대폰/이메일은 nullable 허용 가능하지만,
   UI에서는 둘 중 하나 이상 입력하도록 권장 검증을 넣어주세요.

6. API 설계
==================================================

아래 관리자용 API를 추가해주세요.

[목록]
GET /api/admin/company-contacts
- query params:
  - companyId
  - keyword (담당자명 / 이메일 / 휴대폰 검색)
  - isActive
- 반환:
  - 담당자 목록
  - company_name 포함 가능하면 같이 반환

[단건 조회]
GET /api/admin/company-contacts/:id

[등록]
POST /api/admin/company-contacts
body:
- companyId
- name
- mobile
- email
- department
- position
- officePhone
- memo
- isPrimary
- isQuoteContact
- isBillingContact
- isActive

[수정]
PATCH /api/admin/company-contacts/:id
- 위 필드 수정 가능

[삭제 또는 비활성]
DELETE /api/admin/company-contacts/:id
- 실제 삭제 대신 is_active=false 처리 가능하면 더 좋음
- 이미 프로젝트/견적/청구 이력에 사용된 담당자는 hard delete보다 비활성 권장

7. 기본 담당자 처리 규칙
==================================================

같은 거래처(company_id) 내에서
is_primary=true 담당자는 1명만 존재하도록 해주세요.

즉:
- 새 담당자를 기본 담당자로 저장하면
  기존 기본 담당자는 자동으로 false 처리
또는
- 저장 시 409 에러 + 사용자에게 안내
둘 중 하나로 처리

추천:
- 자동 해제 후 새 기본 담당자로 저장

동일하게,
견적 담당자 / 청구 담당자도
여러 명 허용할지 1명만 허용할지 정책을 분리 가능하게 설계해주세요.

현재 추천:
- is_quote_contact : 여러 명 허용 가능
- is_billing_contact : 여러 명 허용 가능
- is_primary : 1명만 허용

9. 검증 규칙
==================================================

1. 거래처 선택 필수
2. 담당자명 필수
3. 이메일 입력 시 이메일 형식 검증
4. 휴대폰 형식은 너무 엄격하게 막지 말고 기본 문자 검증 수준으로 처리

   - 최소 하나는 입력하도록 권장

10. 삭제/비활성 정책
==================================================

담당자가 과거 프로젝트/견적/청구 이력에 연결될 수 있으므로
삭제는 우선 soft delete 또는 비활성 방식 권장

권장 방식:
- DELETE 요청 시 실제 삭제 대신 is_active=false
- UI에서는 "비활성" 처리
- 필요 시 비활성 담당자도 조회 가능
