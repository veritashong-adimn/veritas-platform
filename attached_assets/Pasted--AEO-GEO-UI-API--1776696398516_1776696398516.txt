지시문

목표:
인사이트 관리 화면에서, 기존 AEO/GEO 강화 작업 결과가 운영자에게도 보이도록 관리자 UI와 API를 확장한다.
이번 작업의 목적은 “검색에 먹히는 구조”를 운영 화면에서도 즉시 판단 가능하게 만드는 것이다.

핵심 방향:
기존 인사이트 관리 테이블 구조는 유지하고, 컬럼 몇 개만 추가해서 가볍게 확장한다.
복잡한 대시보드가 아니라, 리스트에서 바로 AEO 준비 상태를 판단할 수 있게 만든다.

반드시 지킬 원칙:

* 기존 판정(KEEP / REVIEW / MERGE / DROP) 로직은 건드리지 말 것
* 기존 필터/액션 버튼 구조는 유지할 것
* UI 전체 레이아웃 크게 바꾸지 말 것
* 이번 작업은 “관리자 운영 가시성 추가”가 목적이다
* 이미 구현된 공개 페이지 AEO/GEO 로직(slug, faqJson, aeoTitle, aeoDescription, relatedIds, JSON-LD 등)은 재사용하고, 관리자 화면에서 읽어서 보여주기만 하면 된다

작업 범위:

1. 관리자 인사이트 목록 API 확장
   현재 인사이트 관리 목록 API 응답에 아래 파생 필드를 추가한다.

추가 필드:

* faqCount: faqJson 배열 길이
* relatedCount: relatedIds 길이
* hasAeoTitle: aeoTitle 존재 여부(boolean)
* hasAeoDescription: aeoDescription 존재 여부(boolean)
* hasShortAnswer: shortAnswer 존재 여부(boolean)
* aeoStatus: READY | PARTIAL | NONE
* aeoScore: number (0~100)

aeoStatus 계산 기준:

* READY:

  * shortAnswer 존재
  * faqCount >= 3
  * relatedCount >= 1
  * aeoTitle 존재
  * aeoDescription 존재
* PARTIAL:

  * 위 READY 조건을 전부 만족하지 않지만,
  * shortAnswer 또는 faqJson 또는 relatedIds 중 일부라도 존재
* NONE:

  * AEO 관련 값이 사실상 비어 있는 상태

aeoScore 계산 기준:
총 100점

* shortAnswer 있으면 30점
* faqCount >= 3 이면 30점

  * faqCount 1~2 이면 15점
* relatedCount >= 1 이면 20점
* aeoTitle 있으면 10점
* aeoDescription 있으면 10점

주의:

* faqJson이 null/string 혼합 가능성이 있으면 안전하게 파싱 처리
* relatedIds도 null/빈배열 예외 처리
* 점수 계산은 API에서 내려주고 프론트는 표시만 하게 할 것

2. 관리자 인사이트 목록 테이블 컬럼 추가
   기존 인사이트 관리 테이블에 아래 컬럼을 추가한다.

추가 컬럼:

* AEO 상태
* FAQ
* 관련
* AEO 점수

표시 방식:

1. AEO 상태

* READY → 초록 배지
* PARTIAL → 노랑 배지
* NONE → 빨강/회색 배지

문구 예시:

* READY
* PARTIAL
* NONE

2. FAQ

* “FAQ 4”
* “FAQ 0”
  숫자만 보이지 말고 FAQ라는 텍스트 포함

3. 관련

* “연결 3”
* “없음”

4. AEO 점수

* 숫자만 보여도 됨. 예: 90, 55, 0
* 가능하면 80 이상 초록, 40~79 노랑, 39 이하 빨강 계열 텍스트톤 사용
* 과한 그래프는 넣지 말고 심플하게 표시

3. 상세 패널 또는 행 확장 영역에 AEO 요약 표시
   이미 상세 패널이 있다면 아래 정보도 같이 표시한다.

표시 항목:

* AEO Title
* AEO Description
* FAQ 개수
* Related 연결 개수
* AEO 점수
* AEO 상태

값이 없으면:

* “없음”
* “미생성”
  등 명확한 텍스트로 표시

목적:
운영자가 왜 READY/PARTIAL/NONE인지 행 클릭만으로 바로 이해하게 한다.

4. 필터 추가
   기존 필터 바에 아래 필터를 추가한다.

추가 필터:

* AEO 상태 전체

  * 전체
  * READY
  * PARTIAL
  * NONE

가능하면 쿼리 파라미터 연동까지 해라.
기존 서비스유형/상태/공개수준/도메인/언어쌍 검색 구조를 해치지 말 것.

5. 정렬 옵션 추가
   기존 정렬 구조가 있다면 아래 정렬도 지원한다.

* AEO 점수 높은순
* FAQ 많은순
* 관련 연결 많은순

단, 기존 기본 정렬(sourceWeight DESC → filterScore DESC)은 유지하고,
정렬 옵션을 사용했을 때만 바뀌게 할 것.

6. UI 문구/운영 관점
   이번 화면의 핵심은 운영자가 아래를 바로 판단할 수 있게 하는 것이다.

* 이 인사이트는 검색 노출 준비가 되었는가?
* FAQ 보강이 필요한가?
* 관련 인사이트 연결이 부족한가?
* 공개 전에 무엇을 보완해야 하는가?

따라서 문구는 기술적 용어보다 운영 용어에 가깝게 유지:

* AEO 상태
* FAQ
* 관련
* 점수
  이 정도로 단순하게 간다.

7. 구현 시 주의사항

* 기존 판정 배지(REVIEW / MERGE / KEEP 등)와 시각적으로 충돌하지 않게 배지 크기/색상 톤 조절
* 컬럼이 너무 많아지면 폭이 깨질 수 있으니, 새 컬럼은 최대한 짧은 텍스트로 표시
* 반응형에서 너무 무너지면 중요도가 낮은 텍스트는 줄이거나 툴팁 처리
* 기존 살리기 / 병합하기 / 삭제 액션 버튼은 유지
* 기존 테스트 깨지지 않게 할 것

8. 기대 결과
   작업 완료 후 운영자는 인사이트 관리 화면에서 바로 아래를 볼 수 있어야 한다.

* 어떤 인사이트가 AEO 준비 완료인지
* 어떤 인사이트가 FAQ 부족인지
* 어떤 인사이트가 관련 연결 부족인지
* 무엇을 먼저 보완해야 하는지

즉, 화면이 단순 리스트가 아니라 “노출 준비 상태를 관리하는 운영 화면”이 되어야 한다.

9. 완료 보고 형식
   작업 완료 후 아래 형식으로 보고:

* 수정한 파일 목록
* API 추가 필드 목록
* UI 추가 컬럼 목록
* AEO 상태 계산 기준
* AEO 점수 계산 기준
* 테스트 결과
* 캡처 또는 변경 요약
