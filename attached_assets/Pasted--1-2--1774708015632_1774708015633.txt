[목표]
대형 고객(예: 리치몬트코리아) 구조를 반영할 수 있도록
거래처/브랜드/담당자/청구 구조를 분리 설계한다.

핵심은
1) 실제 의뢰 주체
2) 세금계산서 발행 대상
3) 실제 입금 주체
를 서로 다르게 설정할 수 있게 만드는 것이다.

--------------------------------------------------
[실제 업무 예시]
리치몬트코리아의 경우:

- 리치몬트코리아 본사에서 직접 통번역을 의뢰하는 경우가 있음
- 반클리프 아펠, 몽블랑, 까르띠에 같은 브랜드사에서 통번역을 의뢰하는 경우도 있음
- 하지만 세금계산서는 리치몬트코리아 사업자 앞으로 발행
- 실제 입금도 리치몬트코리아가 함

즉,
의뢰 주체와 청구/입금 주체가 다를 수 있음

--------------------------------------------------
[핵심 설계 원칙]

기존 단순 구조:
company → contacts

이 구조로는 부족하므로 아래처럼 분리

Company (법인/청구 주체)
  ↓
Division / Brand (실제 의뢰 조직)
  ↓
Contact (담당자)

그리고 프로젝트에는 다음 관계를 별도로 저장

- requesting_company_id      : 의뢰 상위 법인
- requesting_division_id     : 실제 의뢰 브랜드/부서
- billing_company_id         : 세금계산서 발행 대상 법인
- payer_company_id           : 실제 입금 주체 법인

보통 리치몬트 사례에서는
- requesting_company_id = 리치몬트코리아
- requesting_division_id = 까르띠에 / 몽블랑 / 반클리프 등
- billing_company_id = 리치몬트코리아
- payer_company_id = 리치몬트코리아

--------------------------------------------------
[1. DB 구조 추가/수정]

1) companies 테이블
- 기존 유지
- 법인/사업자 단위

예:
- 리치몬트코리아

2) divisions 테이블 신설
fields:
- id
- company_id (FK → companies.id)
- name
- type (brand / department / team 등, optional)
- created_at

예:
- company_id = 리치몬트코리아
- name = 까르띠에
- name = 몽블랑
- name = 반클리프 아펠

3) contacts 테이블 수정
추가:
- division_id (nullable FK → divisions.id)

의미:
- 브랜드 담당자면 division_id 있음
- 본사 공통 담당자면 division_id nullable 가능

4) projects 테이블 수정
기존 company/customer/admin 구조 외에 아래 컬럼 추가

- requesting_company_id nullable
- requesting_division_id nullable
- billing_company_id nullable
- payer_company_id nullable
- contact_id nullable

설명:
- requesting_company_id: 실제 의뢰가 속한 상위 법인
- requesting_division_id: 실제 의뢰한 브랜드/부서
- billing_company_id: 세금계산서 발행 대상
- payer_company_id: 실제 입금 주체
- contact_id: 실제 요청 담당자

주의:
대부분 일반 고객은
requesting_company_id = billing_company_id = payer_company_id
일 수 있으므로 기본값 자동 복사 가능하게 설계

--------------------------------------------------
[2. 프로젝트 생성/수정 UI 변경]

프로젝트 등록 또는 수정 시 아래 항목을 명확히 분리해서 입력 가능하게 해주세요.

1. 실제 의뢰 조직
- 거래처(법인)
- 브랜드/부서
- 담당자

2. 청구 정보
- 세금계산서 발행 법인

3. 입금 정보
- 실제 입금 주체 법인

UI 동작 원칙:
- 일반 고객은 모두 동일 값으로 자동 채움
- 대형 고객만 필요 시 따로 변경 가능
- 사용자가 복잡하지 않게 기본값 자동 반영

예:
[실제 의뢰]
- 거래처: 리치몬트코리아
- 브랜드: 까르띠에
- 담당자: 홍길동

[청구]
- 세금계산서 발행 대상: 리치몬트코리아

[입금]
- 입금 주체: 리치몬트코리아

--------------------------------------------------
[3. 고객/거래처 상세 화면 변경]

리치몬트 같은 거래처 상세에서 아래가 보여야 한다.

- 법인명
- 하위 브랜드/부서 목록
- 브랜드별 담당자 수
- 본사 공통 담당자
- 브랜드별 프로젝트 수
- 브랜드별 매출
- 전체 미수금 / 선입금 / 누적청구 현황

즉,
리치몬트코리아 상세를 보면
- 까르띠에
- 몽블랑
- 반클리프 아펠
이 보이고,
각 브랜드별 담당자 및 프로젝트가 그룹핑되어 보여야 한다.

--------------------------------------------------
[4. 프로젝트 리스트 / 고객관리 검색 반영]

검색/필터에서 아래 항목을 지원해주세요.

- 상위 법인명
- 브랜드/부서명
- 담당자명
- 세금계산서 발행 대상 법인
- 실제 입금 주체 법인

예:
- "까르띠에"로 검색하면 해당 브랜드 의뢰 프로젝트 조회
- "리치몬트코리아"로 검색하면 본사 의뢰 + 브랜드 의뢰 + 청구 대상 프로젝트까지 조회 가능

--------------------------------------------------
[5. 견적 / 세금계산서 / 결제 반영]

문서와 재무에도 이 구조가 반영되어야 한다.

1. 견적서
- 실제 의뢰 조직(브랜드/부서) 정보 표시 가능
- 필요 시 본사/상위 법인 정보도 표기

2. 세금계산서
- 반드시 billing_company_id 기준으로 발행

3. 결제/수금
- payer_company_id 기준으로 관리

즉,
브랜드사가 의뢰했더라도
세금계산서와 입금은 본사 기준으로 처리 가능해야 한다.

--------------------------------------------------
[6. 기본 UX 원칙]

- 일반 고객은 복잡하게 보이지 않게 한다
- 대형 고객만 확장 구조를 사용하게 한다
- 기본값은 아래처럼 자동 세팅

requesting_company_id = billing_company_id = payer_company_id

- 필요할 때만 사용자가 분리 입력 가능하도록 한다

--------------------------------------------------
[7. 최종 목표]

다음 상황을 자연스럽게 처리할 수 있어야 한다.

예시:
- 까르띠에 담당자가 번역 의뢰
- 프로젝트는 까르띠에 브랜드 프로젝트로 기록
- 하지만 세금계산서는 리치몬트코리아로 발행
- 입금도 리치몬트코리아가 수행

즉,
의뢰 주체 / 청구 주체 / 입금 주체를 분리하여
대형 고객 구조를 안정적으로 처리할 수 있게 개선해주세요.
