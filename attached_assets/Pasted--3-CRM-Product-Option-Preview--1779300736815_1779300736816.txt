상품 마스터 3단계 — CRM 상품 → Product + Option 구조 분석 Preview 작업

중요:
이번 작업은
“실제 상품 구조를 자동 변경하는 작업”
이 아니다.

현재 단계는:

CRM형 상품 구조를
플랫폼 Product + Option 구조로
“분석/추천/preview”
하는 단계이다.

==================================================
핵심 목표
==================================================

현재 CRM 상품은:

- 영한번역
- 한영번역
- 영어동시통역
- 일본어순차통역

처럼:
상품명 안에

- 서비스
- 언어
- 방향

이 모두 들어있는 구조다.

하지만 플랫폼 방향은:

Product
=
실제 서비스 유형

Option
=
언어 / 방향 / 산업 / 난이도

로 분리하는 구조다.

따라서:
Import Preview 단계에서:

“이 항목은 실제 상품인가?”
vs
“옵션으로 분리되어야 하는가?”

를 분석할 수 있도록 한다.

==================================================
절대 금지
==================================================

이번 단계에서는:

- 기존 상품 자동 삭제 금지
- 기존 상품 자동 변환 금지
- 자동 merge 금지
- 기존 products DB rewrite 금지
- quote item 구조 변경 금지
- 프로젝트 구조 변경 금지
- 자동견적 구조 변경 금지
- DB schema 대규모 변경 금지

즉:
이번 단계는
“분석 + 추천 + preview”
까지만 수행한다.

==================================================
1. Product + Option 분석 Preview 추가
==================================================

Import Preview 시:

기존 CRM형 상품명을 분석하여:

- Product 후보
- Language Pair 후보
- Direction 후보

를 분리 분석한다.

예시:

원본:
영한번역

↓

분석 결과:
Product:
번역

Language Pair:
한국어 ↔ 영어

Direction:
ko→en

--------------------------------------------------

원본:
영어동시통역

↓

Product:
동시통역

Language Pair:
한국어 ↔ 영어

Direction:
bidirectional

==================================================
2. Product 추출 규칙
==================================================

상품명에서 실제 서비스 유형을 분석한다.

예시:

- 번역
- 감수
- 원어민감수
- 공증번역

- 동시통역
- 순차통역
- 수행통역
- 화상통역
- 전화통역

- FM 장비
- 리시버
- 부스

등.

--------------------------------------------------

중요:
언어 자체를 Product로 만들지 않는다.

예:
영어동시통역
→ Product:
동시통역

“영어”
는 옵션 후보로 분리.

==================================================
3. Language Pair 분석
==================================================

상품명에서 언어쌍을 추출한다.

예:
- 영한
- 한영
- 일한
- 중한

↓

표준 구조로 normalize:

- 한국어 ↔ 영어
- 한국어 → 영어
- 영어 → 한국어
- 한국어 ↔ 일본어

등.

--------------------------------------------------

중요:
현재 단계에서는:
분석/추천만 수행.

실제 DB 구조 변경 금지.

==================================================
4. Direction 분석
==================================================

Direction은:

- bidirectional
- ko→en
- en→ko

등으로 분석.

--------------------------------------------------

번역(TR):
direction 중요.

예:
- 한국어 → 영어
- 영어 → 한국어

단가/난이도 차이 가능.

--------------------------------------------------

통역(IN):
기본적으로:
bidirectional 추천.

예:
영어동시통역
→ 한국어 ↔ 영어

다만:
향후 글로벌 확장 고려하여:

- ko→en
- en→ko

단방향도 지원 가능하도록 preview에 표시.

==================================================
5. Preview UI 표시
==================================================

Import Preview 화면에 다음 컬럼 추가:

- Product 후보
- Language Pair 후보
- Direction 후보

예:

| 원본 | Product | Language Pair | Direction |
|------|----------|----------------|-----------|
| 영한번역 | 번역 | 한국어 ↔ 영어 | ko→en |
| 영어동시통역 | 동시통역 | 한국어 ↔ 영어 | bidirectional |

--------------------------------------------------

중요:
ERP식 거대한 grid 금지.

현재 compact SaaS preview UX 유지.

↓

표시 가능.

--------------------------------------------------

중요:
자동 통합 금지.
분석/추천만 수행.

10. 최종 확인
==================================================

확인할 것:

1.
영한번역 → 번역 + 한국어↔영어 + ko→en
분석되는지.

2.
영어동시통역 → 동시통역 + 한국어↔영어 + bidirectional
분석되는지.

3.
언어 자체가 Product로 등록되지 않는지.

4.
옵션 후보 표시가 정상 동작하는지.

5.
기존 products 데이터 변경 없는지.

6.
기존 import workflow 정상 유지되는지.

7.
Babel parse / Vite HMR 오류 없는지 확인.
