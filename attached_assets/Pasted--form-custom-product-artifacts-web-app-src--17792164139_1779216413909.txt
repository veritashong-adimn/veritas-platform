견적 수정/재생성 form의 상품 선택 구조를
“커스텀 상품(custom product)” 기반 구조로 개선한다.

대상 파일:
- artifacts/web-app/src/components/admin/ProjectDetailModal.tsx
- 관련 quote item 저장 로직
- 필요한 경우 상품 등록 요청 상태 저장 구조

목표:
현재 “직접 입력”은 상품 DB와 연결되지 않는 자유입력 구조에 가깝다.

이 상태로 두면:
- 상품 통계 누락
- BI 분석 누락
- 반복 사용 패턴 추적 불가
- AI 추천 연결 어려움
- 데이터 자산화 약화

문제가 발생한다.

따라서:
“직접 입력”
이 아니라

“구조화된 커스텀 상품(custom product)”

개념으로 전환한다.

==================================================
1. UI 명칭 변경
==================================================

현재:
- 직접 입력

변경:
- + 커스텀 상품 입력

의미가 더 명확하게 보이도록 한다.

==================================================
2. 커스텀 상품 저장 구조
==================================================

정식 상품 선택 시:

- productId 존재
- isCustomProduct = false

기존 구조 유지.

--------------------------------------------------

커스텀 상품 입력 시:

- productId = null
- isCustomProduct = true

구조로 저장.

다음 정보는 quote item에 구조화 저장:

- customProductName
- serviceType
- languagePair
- quantityUnit
- usagePeriod
- unitPrice
- memo

즉:
자유 텍스트가 아니라
“구조화된 임시 상품 데이터”
로 저장한다.

==================================================
3. 상품 등록 요청 버튼 추가
==================================================

커스텀 상품 입력 후,
직원이 원할 경우
“정식 상품 등록 요청”
을 보낼 수 있게 한다.

예시 버튼:
- 정식 상품 등록 요청

위치는:
- 커스텀 상품 입력 영역 근처
- compact SaaS workflow 유지
- 과도한 버튼 강조 금지

==================================================
4. 상품 등록 요청 저장 구조
==================================================

직원이 요청 시 다음 정보를 저장:

- customProductName
- serviceType
- languagePair
- quantityUnit
- usagePeriod
- unitPrice
- memo
- sourceQuoteItemId
- sourceProjectId
- requestedBy
- requestedAt
- status (pending/approved/rejected)

==================================================
5. 관리자 workflow 방향
==================================================

이번 작업에서는:
“등록 요청 저장”
까지만 구현.

향후:
관리자 상품관리 화면에서:

- 반복 사용된 커스텀 상품
- 등록 요청 목록

을 보고:

- 정식 상품 등록
- 반려

할 수 있게 확장 가능하도록 구조 유지.

==================================================
6. UX 방향
==================================================

중요:
ERP형 상품 등록 프로세스로 만들지 않는다.

즉:
- 별도 큰 modal 금지
- 복잡한 승인창 금지
- table/grid 중심 UI 금지

현재 견적 작성 workflow 안에서
자연스럽게 처리되게 한다.

==================================================
7. 중요한 구조 원칙
==================================================

핵심 방향:

정식 상품
+
커스텀 상품
+
정식 상품 등록 요청

이 하나의 데이터 흐름으로 연결되게 한다.

즉:
“DB 밖 자유입력”
으로 남지 않게 한다.

==================================================
주의사항
==================================================

- 기존 상품 선택 로직 최대한 유지
- 기존 견적 생성/수정/재생성 workflow 깨지지 않게 유지
- DB/API/document service 수정이 필요하면 최소 범위로 진행
- 관련 없는 버그는 수정하지 말고 보고만 한다.
- ERP 느낌 강화 금지
- 과도한 카드화 금지
- vertical spacing 증가 금지
- opacity/transform/blur animation 추가 금지

==================================================
완료 후 확인
==================================================

1. 커스텀 상품 입력이 정상 동작하는지 확인한다.
2. 커스텀 상품이 quote item에 구조화 저장되는지 확인한다.
3. 정식 상품 등록 요청 버튼이 정상 동작하는지 확인한다.
4. 요청 데이터가 정상 저장되는지 확인한다.
5. 기존 정식 상품 선택 workflow가 유지되는지 확인한다.
6. 기존 견적 생성/수정/재생성 기능이 정상 동작하는지 확인한다.
7. Babel parse / Vite HMR 오류 없는지 확인한다.
